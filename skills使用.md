# Aiszr 项目 — Claude Code Skills 中文手册

本文档列举当前可用的所有 Skills，按使用场景分类。每个 Skill 标注触发方式和核心能力。

---

## 一、讨论与设计（Discuss & Design）

### `/grill-me` — 深度方案压测
**做什么：** 追着你逐层追问，把方案的每个分支挖到底，直到没有死角。像面试官一样挑战你的设计决策。
**触发：** 说"压测一下我的想法""这个方案怎么优化""这样设计有什么问题"
**适用：** 大方向面前做决定、拍板前的最后审视

### `/improve-codebase-architecture` — 架构改进
**做什么：** 扫描代码库，找出耦合模块、下沉机会、可抽象重复逻辑，输出改进建议清单
**触发：** 说"帮我看看架构哪能改进""模块太耦合""代码不好维护"
**适用：** 代码越写越乱时，做一次结构性整理

### `/design-an-interface` — API / 接口设计
**做什么：** 启动多个子代理并行设计同一接口的不同方案，生成对比表让你选
**触发：** 说"帮我设计这个 API""这个接口怎么设计""比较两种写法"
**适用：** 跨模块边界定义协议（比如 MuseTalk 子进程通信协议就是典型场景）

### `/doc-coauthoring` — 文档共创
**做什么：** 引导你走完结构化文档创作流程：定目的 → 写初稿 → 逐段打磨 → 读者视角验证
**触发：** 说"写文档""写 ADR""写技术说明"
**适用：** 需要写技术方案、设计决策记录、项目说明

### `/to-prd` — 生成 PRD
**做什么：** 把当前对话上下文综合成产品需求文档，提交为 GitHub Issue
**触发：** 说"生成 PRD""给这个写个需求文档"
**适用：** 把零散讨论变成正式需求

### `/to-issues` — 需求拆 Issues
**做什么：** 把一个大的计划/PRD 拆成独立可领取的 GitHub Issues，按 tracer-bullet 垂直切片
**触发：** 说"帮我拆成 Issues""这个需求可以切几份"
**适用：** 计划确认后，切分任务准备执行

## 二、调试与问题排查（Debug & Investigate）

### `/gsd-debug` — 系统化调试
**做什么：** 按科学方法循环排查：形成假设 → 测试验证 → 修 → checkpoint。状态持久化，跨上下文恢复
**触发：** 说"有 bug""排查问题""帮我调试"
**适用：** 复杂 bug（比如弹幕漏抓、推流断连），不是你一眼能看出来那种

### `/triage-issue` — 问题分诊
**做什么：** 接收 bug 描述 → 探索代码库找根因 → 生成 TDD 修复计划 → 创建 GitHub Issue
**触发：** 说"报 issue""triage""帮我看看这个 bug"
**适用：** 用户用你的软件遇到问题，你要系统化记录并修

### `/qa` — 交互式 QA 会话
**做什么：** 你边测边报问题，我边看代码库理解上下文，帮你提交格式化的 GitHub Issue
**触发：** 说"QA 会话""边测边提 Issue"
**适用：** 正式测试阶段，边用边录问题

### `/gsd-forensics` — 工作流失败分析
**做什么：** 分析一个 GSD 工作流失败的原因，诊断哪里出了问题
**触发：** GSD 命令执行失败后
**适用：** 用 GSD 流程做开发，某个阶段挂了需要复盘

## 三、代码质量（Review & Quality）

### `/review` — PR 代码审查
**做什么：** 对当前分支的变更做标准代码审查：逻辑正确性、安全、性能、可读性
**触发：** 说"review 代码""看看有什么问题"
**适用：** 提交前自查，或 PR 合并前的审查

### `/security-review` — 安全审查
**做什么：** OWASP 对齐的安全审计，重点查：凭证泄露、注入风险、权限校验、敏感数据
**触发：** 说"审查安全""这里会不会有漏洞" 或代码涉及 auth/cookie/credential
**适用：** 涉及登录、API Key、加密、权限等敏感代码

### `/gsd-code-review` — 阶段代码审查（GSD 内）
**做什么：** 审查一个 Phase 中修改的源码，按严重程度分类输出 REVIEW.md
**触发：** 在 GSD 流程中，Phase 完成后自动或手动触发
**适用：** GSD 结构化开发流程里的代码审查环节

### `/simplify` — 代码简化
**做什么：** 读完刚写的代码，检查：有没有重复逻辑？能不能更短？有没有不该抽象的抽象？
**触发：** 超过 3 个文件编辑或 >50 行变更后自动触发
**适用：** 写完功能后看一眼，去肥增瘦

## 四、执行与交付（Execute & Ship）

### `/gsd-quick` — 快速执行
**做什么：** 执行一个小任务，保证原子提交 + 状态追踪，但跳过可选子代理
**触发：** 小修小改，不需要完整 Phase 流程
**适用：** 改个配置、修个拼写、加个简单字段

### `/gsd-execute-phase` — 执行 Phase
**做什么：** 按计划文件的 Wave 分批并行执行，每批原子提交
**触发：** Phase 计划已批准，开始动手
**适用：** 正式开发一个新 Phase（比如我们现在的 MuseTalk 集成）

### `/gsd-ship` — 准备发布
**做什么：** 创建干净的 PR 分支 → 跑 Review → 准备合并
**触发：** 说"ship""create PR""prepare for merge"
**适用：** Phase 完成，准备合并到主分支

### `/gsd-pr-branch` — 创建 PR 分支
**做什么：** 从当前工作过滤掉 .planning/ 提交，创建干净的 PR 分支
**触发：** GSD 流程中，要提交 PR 时
**适用：** 想把代码推给其他人 Review

## 五、TDD 与测试（Testing）

### `/tdd` — 测试驱动开发
**做什么：** 红-绿-重构循环：先写失败的测试 → 写最小代码让它过 → 重构
**触发：** 说"测试驱动""先写测试""TDD""red-green-refactor"
**适用：** 对逻辑清晰的功能写测试先行

### `/gsd-add-tests` — 补充测试（GSD 内）
**做什么：** 根据 Phase 的 UAT 标准和实现生成测试
**触发：** Phase 完成后的补充测试阶段
**适用：** Phase 代码有了但没写测试时

### `/gsd-nyquist-auditor` — Nyquist 测试完整性审计
**做什么：** 检查 Phase 需求的测试覆盖率，补上缺失的测试
**触发：** GSD 流程中，Phase 完成后的验证阶段
**适用：** 确保测试覆盖了所有需求

---

## 六、AI 集成阶段（AI Integration）

### `/gsd-ai-integration-phase` — AI 规范生成
**做什么：** 为包含 AI 组件的 Phase 生成 AI-SPEC.md（框架选择、评估策略、护栏、监控）
**触发：** Phase 涉及 AI/LLM 模型选型或集成
**适用：** MuseTalk 集成、TTS 切换、AI 回复策略调整

### `/gsd-select-framework` — AI 框架选择
**做什么：** 交互式决策矩阵，帮你选出最合适的 AI/LLM 框架
**触发：** 说"选哪个 AI 框架""比较 AI 方案"
**适用：** TTS 切换（IndexTTS vs CosyVoice vs 保留阿里云）这类选型

### `/gsd-eval-review` — 评估审计
**做什么：** 审计已实现 AI Phase 的评估覆盖度，输出 EVAL-REVIEW.md
**触发：** AI Phase 完成后
**适用：** 确保 AI 模块有足够的测试和评估

---

## 七、UI 设计（User Interface）

### `/gsd-ui-phase` — 生成 UI 设计规范
**做什么：** 为前端类 Phase 生成 UI-SPEC.md（设计合约、交互规范、6 维度质量检查）
**触发：** Phase 涉及 UI/前端开发
**适用：** 改数字人页面、加新功能面板等

### `/gsd-ui-review` — UI 视觉审计
**做什么：** 对已实现的前端代码做 6 维度审计（一致性、交互、无障碍、性能、响应式、视觉）
**触发：** 前端 Phase 完成后
**适用：** UI 代码写完后全面检查

### `/gsd-sketch` — UI 草稿原型
**做什么：** 用 HTML 快速出可交互的 UI 草稿，支持迭代
**触发：** 说"画个界面""sketch 一下"
**适用：** 动手写 PyQt 之前先确认交互逻辑

### `/frontend-design` — 前端设计落地
**做什么：** 生产级前端实现（React/Tailwind/shadcn），输出高质量可部署代码
**触发：** 需要 Web 前端界面
**适用：** 非 PyQt 场景（比如 Web 后台管理页）

## 八、工具与环境（Tools & Environment）

### `/setup-pre-commit` — 配置 Pre-commit Hooks
**做什么：** 在当前仓库配置 Husky + lint-staged（Prettier、类型检查、测试）
**触发：** 说"加 pre-commit hook""自动格式化""提交前检查"
**适用：** 多人协作或想规范化提交质量

### `/git-guardrails-claude-code` — Git 安全护栏
**做什么：** 安装钩子阻止危险 Git 操作（push --force, reset --hard, clean, branch -D）
**触发：** 说"保护 git""禁止强制推送""加安全护栏"
**适用：** 防止手滑毁灭性操作

### `/update-config` — 配置 Claude Code
**做什么：** 修改 settings.json：自动行为 hook、权限 allowlist、环境变量等
**触发：** 说"设置权限""自动执行""setting"
**适用：** 调教 Claude Code 本身的行为

### `/gsd-config` — 配置 GSD
**做什么：** 调整 GSD 工作流设置（开关、高级选项、集成、模型配置）
**触发：** GSD 工作流行为需要调整
**适用：** 定制 GSD 流程

### `/keybindings-help` — 自定义快捷键
**做什么：** 设置 Claude Code 的快捷键绑定
**触发：** 说"自定义快捷键""改绑定"
**适用：** 不顺手想改键位

## 九、项目管理（GSD Project Management）

### `/gsd-new-project` — 初始化新项目
**做什么：** 深度收集上下文 + 创建 PROJECT.md + ROADMAP.md
**触发：** 新项目开始
**适用：** 从零启动

### `/gsd-new-milestone` — 新里程碑
**做什么：** 更新 PROJECT.md + 创建新里程碑周期
**触发：** 当前里程碑完成，开始下一轮
**适用：** 版本迭代

### `/gsd-phase` — 管理 Phase
**做什么：** 在 ROADMAP.md 中增删改 Phase
**触发：** 需要调整 Roadmap 结构
**适用：** 重新规划开发顺序

### `/gsd-discuss-phase` — 讨论 Phase
**做什么：** 自适应提问收集 Phase 上下文，准备规划
**触发：** Phase 开始前的讨论
**适用：** 一个功能需求还没想清楚时

### `/gsd-plan-phase` — 规划 Phase
**做什么：** 创建详细 Phase 计划（PLAN.md），含验证循环
**触发：** 讨论清楚后开始规划
**适用：** 正式制定开发计划

### `/gsd-spec-phase` — 澄清 Phase
**做什么：** 用歧义评分评估 Phase 的明确度，产出 SPEC.md
**触发：** Phase 需求模糊需要先澄清
**适用：** 需求不够具体时

### `/gsd-progress` — 进度检查
**做什么：** 查看当前进度、推进工作流、或处理自由格式意图
**触发：** 想看进度、或不确定下一步做什么
**适用：** 随时用

### `/gsd-stats` — 项目统计
**做什么：** 展示项目统计数据（Phase、需求、Git 指标、时间线）
**触发：** 想看整体项目状态
**适用：** 定期检查

## 十、文档与学习（Documentation & Learning）

### `/gsd-docs-update` — 更新项目文档
**做什么：** 根据代码库当前状态生成或更新项目文档
**触发：** 代码变更后需要同步文档

**做什么：** 启动多个并行 Agent 分析代码库，输出结构化文档
**触发：** 需要全面理解代码库
**适用：** 刚接手项目、或用 `/gsd-new-project` 初始化

### `/gsd-extract-learnings` — 提取经验
**做什么：** 从已完成的 Phase 中提取决策、教训、模式、意外发现
**触发：** Phase 完成后做总结
**适用：** 沉淀经验、避免重蹈覆辙

### `/gsd-milestone-summary` — 里程碑总结
**做什么：** 从里程碑产物生成全面的项目总结报告
**触发：** 里程碑结束
**适用：** 团队汇报、给新成员介绍
